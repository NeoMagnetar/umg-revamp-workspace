# BATCH-081 — Compiler / Runtime Extract

## Scope
This extract captures compiler- and runtime-adjacent material from the batch.

## Source-Backed Points
- User assembly of blocks triggers UMG Engine orchestration.
- Generate-button behavior is treated as initiation of the content creation pipeline.
- Block stack is described as executing in priority order.
- Priority toggles are said to influence token weight and execution.
- UMG Engine is described as combining block contents, handling merging, orchestration, and token allocation.
- Recursive execution and modular override behavior are referenced.
- Primary forms the backbone of execution, Merge integrates supporting content, and Blueprint shapes style separately.

## Compiler / Runtime Pressure
- The source implies a deterministic orchestration engine, but does not define the exact normalization, validation, or merge-resolution rules.
- Token allocation and priority execution are mentioned as real runtime surfaces, but without a concrete contract.
- Recursive execution is mentioned, but its scope and boundaries are unspecified.
- The batch supports a cross-batch compiler question:
  - what exact order governs Primary, Merge, Blueprint, Instruction, and Philosophy at runtime?

## Conservative Intake Reading
Useful as evidence that:
- orchestration and priority are recurring concepts
- output shaping is expected to be separated from content payload
- runtime modular override is part of the recurring UMG story

Not enough to:
- finalize compiler law
- define token-weight math
- define merge conflict resolution
- define recursive runtime limits
