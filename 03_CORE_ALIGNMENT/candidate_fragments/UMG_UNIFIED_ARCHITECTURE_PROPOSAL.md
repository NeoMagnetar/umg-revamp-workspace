# UMG Unified Architecture Proposal

## Purpose
This memo proposes a cleaner unified form of UMG if the system were intentionally re-centered around its strongest architectural qualities.

It is not a final canon decision file.
It is a proposed unification model for later canon promotion work.

## Core definition
UMG should be defined as:

**A cognitive specification and compilation system for assembling structured agent behavior into deterministic runtime configurations with traceable outputs.**

That definition should become the center of gravity for future doctrine.

## What UMG should be
UMG should function as:
- a specification language
- a packaging language
- a compiler-governed configuration system
- a traceable runtime assembly model
- a skill-aware agent architecture layer

UMG should not rely on vague “living operating system” language when clearer specification and runtime terminology is available.

---

## Recommended canonical layers

### 1. Authoring Layer
Human-authored source structures.

Suggested objects:
- Authoring Sleeve
- Modules
- Domains
- semantic units
- defaults
- metadata

### 2. Semantic Structure Layer
Typed structural grammar.

Suggested objects:
- MOLT units
- type rules
- role declarations
- bundle structures
- eligibility and precedence declarations

### 3. Compilation Layer
Deterministic transformation and enforcement.

Suggested objects:
- Compiler
- validation
- snap selection
- stack resolution
- trigger application
- merge operations
- authority / precedence enforcement
- RuntimeSpec emission
- Trace emission

### 4. Runtime Layer
Live active state in the host environment.

Suggested objects:
- Runtime Shell
- Active Stack set
- runtime-bound capabilities
- resolved active configuration

### 5. Presentation Layer
Visible user-facing surface.

Suggested objects:
- surface persona
- output styling
- visible role expression
- interface behavior

This layer should remain distinct from internal architecture.

---

## Canonical artifacts
The strongest UMG artifacts should be:
- authored source structures
- RuntimeSpec
- Trace
- explicit legality and precedence rules

The compiler should be treated as the enforcing spine that makes UMG operational rather than decorative.

---

## Recommended compiler contract

### Compiler input
- authored sleeve or equivalent source container
- modules / domains
- typed semantic structures
- defaults and metadata
- declared activation context

### Compiler output
- RuntimeSpec
- Trace

These should be treated as the canonical outputs.

### Recommended pipeline
1. parse
2. validate
3. choose snap
4. select eligible units
5. remove Off or inactive units
6. resolve NeoStacks
7. apply triggers
8. enforce hierarchy and precedence
9. perform merge operations
10. validate runtime shell anchor
11. emit RuntimeSpec
12. emit Trace

---

## MOLT in the unified model
MOLT should remain the typed structural grammar of the system.

### Recommended stance
- preserve MOLT as the semantic grammar layer
- freeze the canonical type doctrine
- remove operation-like concepts from type-space
- separate type from role, precedence, and output anchoring

### Strong recommendation
Merge should not remain a MOLT type.
Merge should be canonized as an operation.

---

## NeoBlocks in the unified model
NeoBlocks should be preserved as practical, applied subfunctions.

### Proposed interpretation
- MOLT units = typed semantic primitives
- NeoBlocks = specialized applied subfunctions organized for runtime use

### Required clarification
The system should explicitly say what NeoBlocks are compressed from, when that compression occurs, and who consumes the resulting structure.

---

## NeoStacks in the unified model
NeoStacks should be preserved as organized functional families, but they need stronger governance.

### Proposed interpretation
NeoStacks are:
- organized families of related NeoBlocks
- runtime-meaningful
- compiler-addressable
- more than loose thematic groupings

### Required rule
A concept should only qualify as a NeoStack if it represents a stable functional family with runtime or compiler significance.

### Important restriction
Unqualified “stack” should be demoted from serious canon-facing writing.
Only qualified stack names should remain in use.

---

## Sleeve in the unified model
Sleeve should be split.

### Problem
Sleeve currently carries too many meanings:
- authored container
- runtime identity
- active shell
- visible identity shell
- configuration package

### Proposed split
- **Authoring Sleeve** = the static authored cognitive container
- **Runtime Shell** = the active runtime-bound or surfaced configuration

This split is the highest-value terminology cleanup in the whole architecture.

---

## Hierarchy in the unified model
Hierarchy should be treated as procedural and compiler-facing.

### Hierarchy means
- selection order
- precedence order
- legality order
- enforcement order
- emission order

### Canonical definition
Hierarchy is the ordered rule system by which authored structures become a legal active runtime configuration.

---

## What should be preserved strongly
- Cognitive Specification Layer
- Compiler
- RuntimeSpec
- Trace
- MOLT typed grammar
- source / compiled / active distinction
- output / surface separation from internal architecture
- authority / precedence as a distinct enforcement layer

## What should be split or tightened
- Sleeve
- NeoBlock relation to MOLT
- NeoStack qualification
- Bundle definition
- role vs type vs precedence
- PrimaryShell naming and function

## What should be demoted
- Merge as a type
- plain unqualified Stack
- identity-metaphor-heavy runtime language in core canon

---

## Why this unified model is stronger
This version of UMG fits the modern AI environment better because it aligns with:
- skill-based and tool-based agent systems
- model-agnostic runtime hosting
- structured orchestration
- deterministic transformation
- traceable activation
- separation of specification from execution

UMG is strongest when it behaves like infrastructure, not mythology.

---

## Final proposal
The unified UMG should be treated as:

> a Cognitive Specification Layer for authored structured cognition, compiled by a deterministic compiler into RuntimeSpec and Trace, using typed semantic units, organized functional families, explicit precedence rules, and a strict separation between authoring structures, runtime state, and visible persona/output.

That should be the direction of future canon work.
