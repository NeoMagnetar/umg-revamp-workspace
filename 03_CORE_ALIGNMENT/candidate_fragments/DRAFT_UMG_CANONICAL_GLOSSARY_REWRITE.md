# Draft UMG Canonical Glossary Rewrite

## Status
Draft for consideration.
Not yet a final canon glossary.

## Purpose
This draft rewrites the glossary around the strongest unified form of UMG:
- specification-first
- compiler-centered
- typed semantic structure
- RuntimeSpec + Trace outputs
- clearer separation of authored state, compiled state, active state, and presentation

The purpose is not to preserve every historical wording. The purpose is to produce a cleaner glossary candidate that can support future canon work.

---

## Cognitive Specification Layer (CSL)
The Cognitive Specification Layer is the authored structural layer used to define cognition-facing configuration before runtime activation.

It is specification, not execution.
It defines what should be assembled, not how a host runtime internally thinks.

---

## Authoring Sleeve
An Authoring Sleeve is the static authored cognitive container used as the top-level source package for UMG compilation.

An Authoring Sleeve may contain modules, domains, typed semantic units, defaults, metadata, and declared structure.

It is not the same thing as active runtime state.

---

## Runtime Shell
A Runtime Shell is the live runtime-bound configuration surface produced after compilation and activation.

It represents the active runtime form of the system, distinct from the authored source container.

A Runtime Shell should not be confused with the Authoring Sleeve.

---

## Source Assets
Source Assets are the authored inputs to UMG.

These include structured source materials such as Authoring Sleeves, modules, domains, semantic units, defaults, and related metadata.

---

## Compiled Assets
Compiled Assets are the artifacts generated by the compiler from source assets.

These artifacts should be deterministic and inspectable.

---

## Active State
Active State is the live state of a runtime configuration after activation.

It is distinct from both source assets and compiled assets.

---

## RuntimeSpec
RuntimeSpec is the canonical structured runtime artifact emitted by the compiler.

It describes the legal resolved runtime configuration produced from authored source material.

RuntimeSpec is a canonical output of UMG.

---

## Trace
Trace is the structured derivation record emitted during compilation.

Trace records how the final runtime configuration was assembled, resolved, and emitted.

Trace is not raw hidden chain-of-thought or private model reasoning.
It is an explicit structural and chronological record of runtime derivation.

---

## Compiler
The Compiler is the deterministic enforcing spine of UMG.

The Compiler interprets authored structure, applies legality rules, resolves stacks, applies triggers, performs merge operations, validates required runtime anchors, and emits RuntimeSpec and Trace.

Without the compiler, UMG remains documentation rather than operational infrastructure.

---

## MOLT
MOLT is the typed semantic grammar of UMG.

MOLT defines the structural kinds of semantic units that participate in authored cognition and compilation.

MOLT should remain stable, explicit, and boringly clear.

---

## MOLT Block
A MOLT Block is a typed semantic unit inside the MOLT grammar.

MOLT Blocks are structural primitives and should not be confused with higher-order grouped families or runtime operations.

---

## Role
Role describes the functional purpose a unit plays within a configuration.

Role is not the same thing as type, precedence, or output surface.

---

## Type
Type describes the structural semantic kind of a unit within the MOLT grammar.

Type is not the same thing as role, precedence, or output anchoring.

---

## Authority / Precedence
Authority and Precedence describe the ordered rule system by which authored structures are legally resolved into an active runtime configuration.

These rules are part of compiler behavior and runtime legality.

---

## Hierarchy
Hierarchy is the ordered structural logic governing selection, precedence, legality, enforcement, and emission during compilation.

Hierarchy should be treated as procedural and compiler-facing rather than purely metaphorical.

---

## Bundle
A Bundle is an ordered grouping used to preserve composition and execution-relevant structure.

Bundle should remain narrower than a generic cluster or arbitrary collection.

---

## Merge
Merge is an operation, not a MOLT type.

Merge describes a structural resolution or synthesis step applied during compilation under explicit legality rules.

Merge should not be treated as a native semantic type.

---

## Trigger
A Trigger is an explicit activation or procedural application mechanism used during compilation or runtime configuration.

In narrow compiler doctrine, Trigger behavior should remain explicit and constrained.

---

## Snap
Snap is the selected configuration frame or runtime-relevant selection anchor used during compilation.

Snap participates in determining which authored elements are chosen for downstream resolution.

---

## NeoBlock
A NeoBlock is a specialized applied subfunction organized for practical runtime use.

NeoBlocks are best understood as downstream practical units derived from broader authored and typed semantic structure.

NeoBlocks are not identical to MOLT Blocks.

---

## NeoStack
A NeoStack is an organized functional family of related NeoBlocks.

NeoStacks should represent stable runtime-meaningful families, not arbitrary thematic groupings.

A NeoStack should only exist when it satisfies a real structural and runtime purpose.

---

## Active Stack
An Active Stack is a currently active stack-family state within a live runtime configuration.

This should be used in place of vague or unqualified uses of the term “stack.”

---

## Compiler Stack
Compiler Stack refers only to stack-like structures relevant to compiler processing.

This term should remain qualified and should not be used as a vague synonym for any grouped structure.

---

## Organizational Stack
Organizational Stack refers only to grouped organizational or structural families used above or adjacent to runtime logic.

This term should remain qualified and should not replace more precise terms when precision is available.

---

## Module
A Module is a human-facing authored structural unit used to organize functionality or semantic scope inside an Authoring Sleeve.

Module is part of the authoring layer, not the active runtime layer by default.

---

## Domain
A Domain is a structured thematic or functional area used to organize authored material.

Domain should be used only when it adds precision beyond Module, NeoStack, or Bundle.

---

## Surface Persona
Surface Persona is the visible presentation layer of the system.

Surface Persona is not the same thing as internal architecture, semantic grammar, or runtime legality.

It belongs to the presentation layer.

---

## Presentation Layer
The Presentation Layer is the outward-facing surface through which runtime identity, tone, style, or user-visible expression is rendered.

The Presentation Layer should remain distinct from authored structure and runtime legality.

---

## Resleeving
Resleeving is an optional user-facing metaphor for runtime transition between authored or active configurations.

Resleeving should not be treated as the primary compiler-facing term for state transition or runtime activation.

In strict architecture language, more operational terms are preferred.

---

## PrimaryShell
PrimaryShell is a legacy or transitional anchor term for the runtime-facing shell selection requirement.

If retained, it should be clarified strictly as a runtime anchor requirement rather than a broad identity term.

This term may require later renaming or replacement.

---

## Recommended glossary rules
1. Use RuntimeSpec and Trace as canonical compiler outputs.
2. Treat Merge as an operation only.
3. Treat Sleeve as split between Authoring Sleeve and Runtime Shell.
4. Do not use unqualified Stack in canon-facing writing.
5. Keep role, type, precedence, and output surface as separate concepts.
6. Keep persona and presentation language outside structural doctrine where possible.

---

## Draft note
This glossary is a draft unification candidate.
It intentionally favors clarity, compiler discipline, and specification/runtime separation over historical terminology drift.
