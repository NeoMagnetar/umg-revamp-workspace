# UMG Canonization Plan — Hierarchy, MOLT, NeoBlocks, NeoStacks

## Purpose
This memo turns the recent architecture review into a practical canonization plan for the parts of UMG carrying the most semantic load and the most contradiction pressure.

It is not a final canon decision file.
It is a working alignment document intended to help move evidence into canon.

## Why this memo exists
The current UMG corpus is carrying multiple doctrinal generations at once. The strongest pressure points cluster around:
- top-down hierarchy and precedence
- MOLT type doctrine
- Merge vs Bundle behavior
- Sleeve meaning
- NeoBlock compression
- NeoStack formation
- compiler-centered runtime assembly

The goal here is to decide what should be preserved, split, renamed, demoted, or formally canonized.

## Recommended high-level stance
UMG should lean harder into its strongest modern form:
- Cognitive Specification Layer framing
- compiler-centered enforcement
- RuntimeSpec + Trace as canonical outputs
- typed semantic units
- explicit separation of source, compiled, and active state

UMG should lean less on older identity-metaphor and all-encompassing “cognitive operating system” framing where that language creates ambiguity.

---

## Canonization recommendations by concept

### 1. Top-down hierarchy
**Recommendation:** Canonize and tighten.

#### Keep
- hierarchy as compiler-relevant ordering
- authority / precedence as explicit runtime resolution logic
- hierarchy as part of deterministic compilation rather than metaphor

#### Clarify
- role vs type vs precedence vs output anchor must be treated as separate axes
- hierarchy should remain structural, not poetic

#### Next action
- create one canon candidate fragment that states the hierarchy only in compilation / precedence language

---

### 2. MOLT blocks
**Recommendation:** Preserve as core doctrine and freeze the type model.

#### Keep
- MOLT as the typed semantic grammar
- block typing as the structural atom layer
- typed ordering as part of compiler legality

#### Tighten
- stop drifting between historical and current type sets
- formalize the exact canonical type set
- remove operation-like concepts from type-space

#### Next action
- prepare one canonical MOLT type memo with explicit “current type set” language

---

### 3. Merge
**Recommendation:** Demote from type-space and canonize as operation.

#### Keep
- Merge as a structural or compiler operation
- same-type merge restrictions where already specified by later compiler logic

#### Remove from central ontology
- Merge should not remain half-in / half-out as a canonical MOLT type

#### Why
Treating Merge as a type weakens grammar clarity, compiler law, and trace semantics.

#### Next action
- create a decision note: “Merge is an operation, not a MOLT type.”

---

### 4. Bundle
**Recommendation:** Preserve, but define narrowly.

#### Keep
- Bundle as ordered grouping
- bundle as execution-chain or composition grouping language

#### Avoid
- letting Bundle blur into generic cluster / container language

#### Next action
- define Bundle in relation to MOLT blocks, NeoBlocks, and execution order

---

### 5. Sleeve
**Recommendation:** Split into two canonical layers.

#### Problem
Sleeve currently carries too many meanings at once:
- authored container
- runtime identity
- active shell
- visible identity
- resleevable configuration package

#### Proposed split
- **Authoring Sleeve** = static authored cognitive container
- **Active Sleeve / Runtime Shell** = live runtime-bound or surfaced configuration state

#### Why
This is the single biggest terminology cleanup that would reduce downstream ambiguity.

#### Next action
- draft a sleeve split memo showing authored state vs active runtime state

---

### 6. NeoBlocks
**Recommendation:** Preserve, but explain compression more explicitly.

#### Keep
- NeoBlocks as specialized subfunctions
- NeoBlocks as practical decompositions under higher-order stack families

#### Clarify
- what they are compressed from
- what stage they belong to
- how they differ from MOLT blocks

#### Best current interpretation
- MOLT blocks = typed semantic structural units
- NeoBlocks = specialized applied subfunctions used within organized runtime families

#### Next action
- author a short note: “NeoBlock compression and its relationship to MOLT blocks”

---

### 7. NeoStacks
**Recommendation:** Preserve as organized families, but enforce stronger governance.

#### Keep
- NeoStacks as grouped higher-order families
- stack resolution as compiler-relevant behavior

#### Risks
- unbounded stack proliferation
- overlap with roles, modules, or metadata
- confusion with generic software “stack” language

#### Rule to introduce
A concept should not become a NeoStack unless it satisfies a defined stack criterion:
- persistent family role
- multiple subordinate subfunctions
- compiler- or runtime-relevant grouping logic
- stable enough to justify stack identity

#### Next action
- create a NeoStack qualification rule so new stacks are not added casually

---

### 8. Stack (unqualified)
**Recommendation:** Demote as a standalone canonical noun.

#### Keep only when qualified
- Compiler Stack
- Organizational Stack
- Active Stack
- NeoStack

#### Why
Unqualified “stack” is already overloaded inside UMG and in the wider software / AI ecosystem.

#### Next action
- add a glossary rule: use qualified stack names only in canon-facing writing

---

### 9. Compiler
**Recommendation:** Elevate as the enforcing spine of modern UMG.

#### Keep
- compiler as deterministic interpreter of CSL
- RuntimeSpec + Trace as canonical outputs
- explicit stage ordering
- legality checks around stack resolution, triggers, merge, and PrimaryShell

#### Why
This is the strongest maturation point in the corpus. It is where UMG becomes more than ontology.

#### Next action
- promote compiler contract language into canonical audit and decision logs

---

### 10. Trace
**Recommendation:** Preserve strongly and protect its meaning.

#### Keep
- Trace as structural, chronological derivation record
- Trace as emitted artifact tied to RuntimeSpec formation

#### Avoid
- treating Trace as hidden raw chain-of-thought exposure

#### Next action
- define a canon-safe trace taxonomy and event model later

---

## Canonization queue

### Immediate queue
1. Merge is an operation, not a MOLT type.
2. Sleeve split: authored container vs active runtime shell.
3. Freeze current MOLT type set.
4. Canonize compiler outputs: RuntimeSpec + Trace.
5. Require qualified use of “stack.”

### Short-follow queue
6. NeoBlock compression memo.
7. NeoStack qualification rule.
8. Bundle definition memo.
9. role vs type vs precedence note.

### Later queue
10. Neo* public-vs-internal vocabulary review.
11. PrimaryShell naming cleanup.
12. trace event taxonomy.

---

## Suggested canon labels
- **Preserve:** MOLT block, RuntimeSpec, Trace, compiler-centered hierarchy
- **Split:** Sleeve
- **Demote:** unqualified Stack, Merge-as-type
- **Tighten:** NeoBlock, NeoStack, Bundle, Role/Type distinction
- **Canonize soon:** CSL framing, deterministic compiler contract

---

## Final recommendation
UMG should continue evolving away from broad identity-metaphor architecture and toward:
- specification clarity
- compilation discipline
- traceable assembly
- skill- and runtime-friendly structure

That is where the system is strongest, most modern, and most likely to survive contact with real AI tooling.
