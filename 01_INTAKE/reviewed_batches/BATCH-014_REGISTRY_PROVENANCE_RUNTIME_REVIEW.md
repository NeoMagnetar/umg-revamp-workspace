## UMG-Relevant Extraction
### 1. Core semantics reinforced
- Block = atomic unit of meaning, cognition-oriented, identity-stable across composition.
- NeoBlock = compiled active view of blocks inside a stack.
- NeoStack = domain-level aggregation of NeoBlocks; explicitly activated; ON/OFF capable.
- Sleeve = what the system is right now; contains governance, active stacks, defaults, and constraints.
- Determinism remains a standing requirement.

### 2. MOLT and request parsing
- File-backed MOLT role set remains `TR / DIR / INS / SUB / PRI / PHI / BP`.
- Request interpretation begins with parsing a short MOLT skeleton.
- Runtime materials extend MOLT into named atoms embedded inside ordered NeoStacks and NeoBlocks.

### 3. Mode activation and NeoStack expansion
- Sleeve mode activation can be triggered by upload or request.
- Activation protocol requires enumerating NeoStacks, generating required NeoBlocks, optional bundles, and emitting full sleeve structure in the first response.
- The 60-mode catalog is structurally valuable as a mode/NeoStack registry, but not obviously identical to a full canonical sleeve-ID registry.

### 4. Runtime / bounded agency bridge
- RuntimeSpec adds typed contracts, ordered phases, bundle policy, anti-thrash lock, risk/alignment gating, governance lock, drift thresholds, and audit traces.
- Safety layer frames the runtime as deterministic, policy-driven, and bounded.
- This batch therefore bridges sleeve-level compositional control and a more formal auditable runtime direction.

### 5. Provenance pressure
- `SLV.POE_UMG.MAIN.v1` is clearly file-backed in surfaced material.
- Additional sleeve registries rendered in the chat were useful, but not fully corroborated by the surfaced excerpts.
- Provenance labeling is therefore a first-class project need.
