## UMG-Relevant Extraction

### Core UMG Semantics
UMG was described publicly as:
- not just an app
- not just a website
- not just a productivity tool
- a cognitive framework
- a novel paradigm
- a system centered on behavioral composition
- reusable thought rather than only reusable code
- tech-agnostic and broadly applicable

The chat explicitly contrasts:
- other systems extend capability
- UMG composes behavior

A useful public comparison preserved in the source:
- other systems: connect to a database
- UMG: think about data ethically and efficiently

### MOLT Roles / Taxonomy
UMG was described as being built from atomic logic units called MOLT blocks.

Explicit public role definitions in the source:
- Primary = goal / intent
- Subject = domain
- Instruction = logic
- Philosophy = ethics and tone / style and worldview
- Blueprint = style / format / form / presentation

Later revisions added that blocks carry:
- functional logic
- embedded metadata
- contextual alignment
- structured data
- merge logic

The pitch layer is simplified and presentation-ready rather than a full formal taxonomy.

### Merge
The source states:
- blocks snap into stacks
- stacks merge into sleeves
- merge hierarchy governs behavioral flow
- UMG uses a merge engine

Merge also appears in builder actions:
- merge multiple plans into one
- inject overlays for compliance or tone
- semantic overlays
- composable cognition

This is strong public-facing language, but operational merge law is not formally defined here.

### NeoBlock Composition
The chat does not use the term NeoBlock directly, but it strongly supports block-style composition.

Blocks are described as:
- atomic logic units
- role-defined
- editable
- draggable
- snap-together planning units
- metadata-bearing
- adaptive across systems

The builder framing strongly treats blocks as modular planning objects that can encode both meaning and operational context.

### NeoStack Architecture
The chat does not use the term NeoStack directly, but it does describe stack architecture clearly:
- blocks form stacks
- stacks form sleeves
- stack hierarchy governs behavioral flow

The builder output is described as a modular logic plan broken into blocks that can be snapped and merged.
Stack hierarchy is also used in the interoperability framing.

### Sleeve Identity / Resleeving
Sleeves are explicitly mentioned as the layer above stacks.
No explicit sleeve identity model, sleeve naming rule, or resleeving protocol is defined here.
Sleeves function mainly as part of the public hierarchy explanation.

### Governance / Control
Ethics and alignment are central to the public explanation.
Philosophy blocks are described as carrying:
- ethics
- tone
- worldview

The builder is described as being able to add:
- ethical overlays
- compliance overlays

Blocks are also said to carry contextual alignment.
UMG is positioned as improving how systems behave, not only what they can do.

### Output Blueprint / Rendering Separation
Blueprint blocks are defined as shaping:
- style
- format
- form
- presentation

The source uses a public analogy:
- Blocks = HTML (structure)
- Philosophy = CSS (style/tone)

This preserves a public-facing separation between structure, style, and behavior.
The builder is framed as a semantic planning engine rather than only a rendering interface.

### Compiler / Runtime Logic
The builder is described as taking a natural-language goal and returning a modular logic plan.

Example preserved in the source:
- build a support chatbot for a crypto wallet
- output: tech stack, user flow, ethics, UI, broken into editable blocks

Runtime-like builder actions in the source:
- snap new logic blocks into place
- inject overlays for compliance or tone
- merge multiple plans
- export as agent logic, apps, or documentation

No formal compiler pipeline or runtime activation model is defined here.
The value is product-demo level explanation.
