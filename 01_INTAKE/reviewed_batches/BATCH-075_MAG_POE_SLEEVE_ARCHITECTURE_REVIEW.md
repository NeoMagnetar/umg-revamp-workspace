- Blocks are modular units of cognition.
- NeoBlocks are structured blocks with stable fields.
- NeoStacks are grouped cognitive layers.
- Sleeves are full cognitive configurations composed of NeoStacks.
- GPT-4o substrate, PoeUMG overlay, fusion runtime, and higher layers are explicitly separated.

- MOLT fields explicitly named: Trigger, Directive, Instruction, Subject, Primary, Philosophy, Blueprint, Bundle, Merge.
- Trigger is treated as a gate rather than a stack element.
- Bundle is treated as grouped deterministic block sets.
- Merge is treated as deterministic combination logic tied to weight and trigger conditions.

### Weight and control model
- Priority is expressed through a numeric weight scale from immutable core to dormant.
- Layer order is explicitly described as Core Identity > Time Depth/Horizon > Strategy > Tone.
- Trigger escalation changes weights dynamically.

### NeoStack and sleeve architecture
- Layered architecture includes substrate, overlay, fusion runtime, strategic persona, canon enforcement, utility extensions, mythic layer, and Genesis/meta creation layer.
- Sleeve identity is versioned and explicitly revisioned across releases.
- Resleeving is implied through fusion controls, sleeve control blocks, and Genesis mutation capability.

### Runtime and compiler-adjacent signal
- Fusion Runtime is described as substrate draft -> merge -> reflection -> final render.
- Genesis acts as block compiler / mutation engine / registry.
- Context injection bridges runtime and stored MOLT structures.
- Utility layer supports parsing, context stitching, and reflection hooks.

### Governance and safety
- Canon enforcement layer contains non-negotiables and conflict order.
- Override resistance and safety interlock logic are described.
- Safety includes no coercion, no exploitation, and human-first alignment.
- Simulation² is presented as a governing framing boundary.

## Independent Review
This batch is valuable because it does not stop at isolated role language. It proposes an integrated sleeve system linking schema, runtime, safety, versioning, and release logic. That makes it stronger than a purely conceptual positioning batch. It also creates a distinct sleeve-specific evidence surface that was not fully present in earlier UI, docs, or comparative batches.

At the same time, the source remains provisional. Several elements read like internal system doctrine but are not backed here by a raw transcript or a finalized canonical decision. In particular, Genesis authority is expansive, the spatial MOLT model is only partially defined, and the mythic layer is present without a hard control boundary. The correct Stage 1 treatment is therefore evidence preservation plus targeted extraction, not canon promotion.
