# BATCH-156 Canon Candidates

## Status
These are candidate canon statements extracted from the source. They are **not approved** and should remain evidence-stage only.

## Candidate Statements

### Core definitions
- Block = atomic logic or asset unit stored as JSON.
- Stack = vertical chain of blocks.
- Sleeve = operational persona composed of one or more stacks.
- Snap = compatibility binding for automatic attachment.

### Composition
- Compatible blocks snap into provisional structures before merge resolution finalizes the usable stack.
- Merge logic is a distinct composition stage and should not be conflated with snap detection.

### Schema
- A block may carry semantic metadata, composition metadata, display metadata, governance metadata, runtime metadata, and optional executable module references.

### Compiler/runtime
- Blocks may reference code modules through a formal attachment surface such as `code_modules`.
- Public compiler claims should be separated into implemented, planned, and speculative capability.

### Documentation / repo framing
- Public-facing block documentation should distinguish current capability from long-range ambition.
- Any narrative connecting block JSON to executable kernels requires an explicit implementation-status matrix.

## Notes
Several strong phrases from the batch are reusable for documentation, but many are still too aspirational for canon without implementation backing.
