  - personality
  - professional
  - abstract / bizarre
- Each sleeve is mapped to **four NeoStacks**.
- The same source also describes a runtime activation protocol where selecting a sleeve should trigger stack enumeration, downstream NeoBlock generation, optional bundle addition, and sustained in-sleeve behavior until explicit switch.

## Chat-Level Summary
This batch is primarily a taxonomy-and-runtime batch. It does two things at once:
1. expands sleeve-space substantially through a structured 60-sleeve catalog, and
2. proposes a deterministic sleeve activation doctrine for later runtime use.

The strongest value is not just the names. The stronger value is the claim that sleeves are profileable identity artifacts with stable governance anchoring and a predictable first-response expansion pattern.

## UMG-Relevant Extraction
### Core semantics
- Sleeves are treated as modular identity configurations composed from NeoStacks.
- The catalog is divided into three sleeve families rather than a flat list.
- The source explicitly frames this as identity architecture, not just style labels.

### Sleeve taxonomy
- 20 personality sleeves
- 20 professional sleeves
- 20 abstract / bizarre sleeves

### NeoStack architecture
- Every sleeve receives four NeoStacks.
- This creates a normalized catalog structure at the sleeve -> NeoStack layer.
- NeoBlock expansion is deferred rather than fully authored here.

### Activation doctrine
- On mode selection:
  1. activate selected sleeve
  2. enumerate NeoStacks
  3. generate primary NeoBlock per stack
  4. generate instruction NeoBlocks
  5. add optional bundles
  6. emit full structure in first response
  7. remain in sleeve until explicit switch

### Governance / control
- Governance remains anchored to `POE.UMG.CORE`.
- Switching is treated as deterministic and explicit.
- Sleeve state is expected to persist until an explicit switch command is issued.

### Output / rendering separation
- Structured sleeve catalog content is distinct from the requested PDF artifact.
- The PDF is a rendered documentation object, not the underlying canonical structure.

## Independent Review
This is a strong Stage 1 evidence batch for sleeve-catalog architecture and activation semantics.

Its strongest contribution is the **activation doctrine**, not the raw count of modes. The source effectively proposes a runtime contract for sleeve switching and first-response expansion. That is materially useful to:
- sleeve architecture documentation,
- runtime-controller planning,
- future compiler work around activation and persistence.

The batch is weaker at lower-layer detail. NeoBlocks are not actually expanded for the full catalog, and no compatibility/conflict system is applied yet. Because of that, the batch should be preserved as evidence for taxonomy and activation protocol, not promoted directly into canon.
