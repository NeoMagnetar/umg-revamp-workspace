## Chat-Level Summary
This batch positions UMG as a model-agnostic modular cognition architecture with governance as a first-order control layer. It defines MOLT as the atomic unit, describes sleeves and NeoStacks as runtime containers, proposes invocation-based activation, and frames continuity across model transitions as a core design principle. It also pushes UMG toward a compiler/runtime split through stack loading, render flow, and structured output envelopes, while leaving merge direction, hierarchy law, and symbolic-versus-executable governance boundaries unresolved.

## UMG-Relevant Extraction

### Core semantics
- UMG is defined as Universal Modular Generation and treated as a modular thought architecture.
- The framework is described as model-agnostic and intended to persist across model changes.
- Intent, context, philosophy, and instruction are separated into discrete units.
- “Structure survives upgrades” is presented as a core continuity principle.

### MOLT / taxonomy
- MOLT is treated as the atomic unit of meaning, instruction, logic, and style.
- Roles explicitly named: Primary, Subject, Philosophy, Instruction.
- MOLT can be created, merged, layered hierarchically, and used as directive or structural unit.
- “Active Stack” is treated as the current bundled execution context.

### Trigger / invocation
- Invocation is treated as the activation entry point.
- Trigger appears as a field in the MOLT Map.
- Activation phrases and explicit stack-load language imply a runtime gate rather than passive mention.

### Priority / governance
- Governance is declared to precede all layers.
- Conflict ordering is explicit: DEFENSE > DISARM > ESCAPE > STRIKE_LAST.
- Legacy hierarchy is also described: LOYALTY > CODE > TRUTH.
- Governance blocks are treated as always-on control surfaces once loaded.

### Merge / bundle
- MOLT blocks can merge into composite logic.
- Merge directionality is mentioned as important but unresolved.
- NeoStacks function as bundle-like grouped execution structures.

### NeoBlock / NeoStack / sleeve structure
- Canonical naming system includes NB.*, SLV.*, NSTK.*.
- NeoBlocks contain identity, philosophy, response style, conflict order, and mythos.
- Sleeves are behavior/persona layers and can move across models.
- “Envoys” suggest different models loading the same sleeve.
- ActiveStack is treated as a runtime snapshot with fields like Sleeve, Mode, Objective.

### Blueprint / rendering separation
- Output envelope is highly structured.
- Declarative block content is separated from rendered prose.
- Blueprint versus execution is explicitly distinguished.

### Compiler / runtime
- UMG is described as a compiler for structured prompts.
- Runtime is framed as ActiveStack + MOLT Map execution.
- Invocation -> Stack load -> Output render is the implied flow.
- Cross-session reconstruction from grammar is treated as a continuity mechanism.

### Skills / workflow / release
- Integration with OpenClaw / ClawHub skills is discussed.
- Slash invocation model is proposed.
- Workspace files are used for persistence.
- GitHub repo, Replit prototype, whitepaper, and outreach imply early release/staging surfaces.

## Independent Review
This is one of the stronger continuity-and-governance batches so far. It materially advances the project in three ways. First, it pushes UMG above any single model by making portability and re-sleeving explicit. Second, it treats governance as a loaded runtime layer instead of a loose value statement. Third, it sharpens the compiler/runtime separation enough to justify targeted extraction for later synthesis.

At the same time, the source remains a summary. That matters. Some statements read like settled doctrine while others remain clearly symbolic, interpersonal, or aspirational. The batch is therefore best treated as strong evidence for later synthesis, not as direct canon.
