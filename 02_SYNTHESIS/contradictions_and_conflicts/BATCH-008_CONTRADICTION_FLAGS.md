# BATCH-008 — Contradiction Flags

## Flag 001 — “Always enforced” governance vs hosted-platform limits
- **Tension:** governance language is stated as always enforced and supreme, but the batch also clarifies that user-authored instructions cannot supersede the host platform.
- **Why it matters:** without separating symbolic authority from executable runtime authority, UMG claims can overstate what hosted implementations actually do.
- **Status:** unresolved in this batch; needs explicit control-layer documentation.

## Flag 002 — Memory-level canon vs real persistence
- **Tension:** governance language is requested/stored as memory-level canon, but the batch clarifies memory is not the same as immutable runtime law.
- **Why it matters:** persistence language may be misread as stronger than the platform actually supports.
- **Status:** unresolved; should be formalized in memory/process-law documentation.

## Flag 003 — Restart language vs literal reset behavior
- **Tension:** “restart simulation” sounds operational, but the batch clarifies it is conversational shorthand rather than literal runtime reset.
- **Why it matters:** restart, reset, and resleeve need formal semantics to avoid ambiguity.
- **Status:** unresolved; candidate runtime contract task.

## Flag 004 — QR/sigil gate as symbolic vs executable
- **Tension:** the sigil/QR artifact is treated as a control condition in canon language, but is not operational in the hosted environment.
- **Why it matters:** symbolic gate objects should not be confused with actual runtime gate specs before owned runtime exists.
- **Status:** unresolved; candidate migration path to future runtime gating.
