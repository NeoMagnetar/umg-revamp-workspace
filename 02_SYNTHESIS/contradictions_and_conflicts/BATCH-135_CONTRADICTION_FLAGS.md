# BATCH-135 — Contradiction Flags

## Scope Note
These are not canon decisions.
They are tensions or possible conflicts surfaced by the source.

## Flag 1
**Topic:** Generic template contamination  
**Tension:** Reusable scaffolds are useful, but the user repeatedly rejected semantically alien sections imported from other blocks.  
**Why it matters:** The project needs a stable rule for minimum scaffold versus block-native structure.

## Flag 2
**Topic:** Exact visual fidelity in plain chat  
**Tension:** The user wanted exact visual replication of a header/card style, but the chat concluded that typography fidelity belongs to a renderer layer.  
**Why it matters:** Semantic render contracts and renderer specifications must remain distinct.

## Flag 3
**Topic:** Recursive decomposition depth  
**Tension:** The source strongly implies fractal recursion into schemas, validators, and traces, but it does not finalize a universal stopping rule.  
**Why it matters:** Future compiler or documentation work may need explicit guidance on when decomposition is required versus optional.

## Flag 4
**Topic:** Illustrative backend forms versus canon syntax  
**Tension:** The chat generated code-like backend/compiler examples, but they were illustrative rather than declared canonical syntax.  
**Why it matters:** Later implementation should not assume every illustrative tree or manifest is already a locked standard.

## Flag 5
**Topic:** Supporting uploads versus ratified outputs  
**Tension:** Uploaded roadmap and glossary materials aligned strongly with the chat but were not explicitly ratified in the live conversation as merged block law.  
**Why it matters:** Supporting artifacts remain evidence inputs rather than automatic canon.
