# BATCH-156 Definitions Extract

## Candidate Definitions from Source
These are evidence-stage definitions extracted from the batch. They are not canon.

### Block
Atomic logic or asset unit stored as JSON; described as self-describing, composable, executable, and portable.

### Stack
Vertical chain of blocks; hierarchical composition.

### Sleeve
Operational persona or runtime composition built from one or more stacks.

### Snap
Compatibility binding or attachment rule between blocks.

### Merge Engine
System that resolves conflicts and finalizes stack composition.

### Cantocore
Human-readable or poetic control language embedded in blocks.

### Mojo
High-performance language intended to compile block logic into kernels.

### Max Compute
Proposed build pipeline for compiling Mojo modules into binary/WASM artifacts.

### Trigger
Runtime activation condition for mutation or stack rewriting; also presented as the highest-priority layer in merge resolution.

### Priority
Ordered merge precedence participating in conflict handling and automatic composition order.

### Bundle (`.sleeve`)
Packaging stage positioned after compilation and binary sealing in the proposed pipeline; technically undefined in this batch.

### Guardrail
Block-local control or safety construct used to restrict override or force safe-state behavior.

### VSS / `RETURN:SAFE.STATE`
Runtime failure or containment language indicating fallback to a safe state; not fully defined in this batch.

### Declarative Canvas
Claimed UI/rendering surface driven by declarative block metadata rather than traditional frontend assembly; not fully defined in this batch.

## Candidate Structural Terms
### `molt_type`
Core classifier used to identify block role/type in the schema example.

### `snap_config`
Schema area describing attachment compatibility and override constraints.

### `merge_logic`
Schema area describing priority, dependencies, conflict behavior, and merge order.

### `code_modules`
Schema area connecting block data to executable code artifacts.

### `integration_layer`
Draft field indicating where the block participates in system execution.

### `agent_orchestration`
Draft field indicating agent execution metadata such as executor type and required envoys.

## Caution Notes
- Definitions above were surfaced in a draft/documentation-oriented conversation.
- Several terms are only partially specified.
- None of these definitions should be treated as finalized canon from this batch alone.
