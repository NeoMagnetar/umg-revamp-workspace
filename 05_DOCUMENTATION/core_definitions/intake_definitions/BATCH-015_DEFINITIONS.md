# BATCH-015 — Definitions Extract

## Terminology
### Governance
Stable upstream control layer. Not owned by output scaffolding. Not a project-visible style rule.

### Sleeve
What the system is right now. Contains governance anchor relationship, active stacks, defaults, and constraints.

### NeoStack
Domain-level cognition or work mode. Explicitly activated and visible in Active Stack if active.

### NeoBlock
Compiled active view of blocks within a stack. Supports one Primary, layered Instructions, and optional bundle effects.

### Block
Atomic unit of meaning.

### MOLT Map
Declarative snapshot of current interpretive structure:
- Trigger
- Directive
- Instruction
- Subject
- Primary
- Philosophy
- Blueprint

Not hidden chain-of-thought.

### Active Stack
Visible runtime truth surface. Candidate rule from batch: if it is not shown in Active Stack, it is not active.

### MODE
How the system thinks.

### BP_MODE
How the system emits output.

### Section 4 / Formal Response Content
The user-facing answer body. Candidate rule from batch: it should behave as if the envelope does not exist.

## Role Language
- Trigger = request/initiation context
- Directive = what must happen
- Instruction = how to proceed
- Subject = topical lens
- Primary = dominant function
- Philosophy = stance / worldview
- Blueprint = output shaping only

## Architecture Language
- System = stable governance/honesty/non-execution core
- Project = sleeve/stack/runtime posture and visible envelope
- Preferences = style only
- file-backed knowledge = libraries/specs/reference material

## Runtime / Compiler Language
- request parsing
- stack selection
- governance application
- blueprint selection
- output emission
- bounded self-improvement
- governance lock
- anti-thrash lock
- drift thresholds
- audit trace
