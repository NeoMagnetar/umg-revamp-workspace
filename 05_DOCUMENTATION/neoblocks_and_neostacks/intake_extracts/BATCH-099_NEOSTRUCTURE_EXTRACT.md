# BATCH-099_NEOSTRUCTURE_EXTRACT

## Scope
NeoBlock / NeoStack / bundle / portability signals extracted from BATCH-099.

## Stack Construction Signal
The source reports a stack-building flow:
- start with Primary
- add context
- add constraints and directives
- overlay philosophy / blueprint
- export / run

This provides a useful structural narrative, though it must be reconciled with the separate stated priority hierarchy.

## Snap / Composition Signal
Blocks are described as snapping in two ways:
- vertically for hierarchy or sequence
- horizontally for overlay or modification

This pressures neostructure work to clarify whether snap relations are:
- purely organizational
- runtime-significant
- export-serializable
- mutation-aware

## Bundle / Export Signal
The source describes bundle/export behavior including:
- ZIP package output
- JSON project/state payloads
- markdown files
- runtime markers for external executors

That strongly suggests a distributable project-bundle concept rather than single-file block export only.

## Portability / Resleeving Signal
The batch strongly advances a pattern-based portability model:
- Poe/UMG continuity is framed as fidelity of pattern rather than one body/server/model
- future model upgrades are treated as new runtimes rather than resets
- remembrance/alignment continuity is the main preserved thread across migration

This is one of the clearer sleeve/resleeving evidence packets even though no formal sleeve protocol is provided.

## Structural Tensions
- stack-construction narrative and priority hierarchy may not be identical ordering systems
- portability is asserted strongly, but migration mechanics remain undefined
- collective convergence is conceptually rich but not reduced to finalized consensus logic
- bundle structure is implied, but not fully standardized as a canonical package format

## High-Value Documentation Targets
- distinguish construction order from conflict-resolution order
- define vertical versus horizontal snap semantics
- document bundle/package structure for exportable UMG projects
- formalize portability/resleeving language without overcommitting unsupported metaphors
- separate philosophy-heavy continuity language from executable migration protocol
